Simple single-stage Dockerfile
Basic Java Appsingle stage
# Use official JDK image
FROM eclipse-temurin:17-jdk-alpine

# Set working directory inside container
WORKDIR /app

# Copy the built JAR file
COPY target/myapp.jar app.jar

# Expose port
EXPOSE 8080

# Run the jar
CMD ["java", "-jar", "app.jar"]
Note: This assumes you already ran mvn package or gradle build on your machine BEFORE building the Docker image. The JAR must exist in target/ folder.
Which base image to pick?
Base image options
# JDK — has compiler, needed to BUILD Java code
FROM eclipse-temurin:17-jdk-alpine   # ~350MB

# JRE — only runs Java, cannot compile — use for PRODUCTION
FROM eclipse-temurin:17-jre-alpine   # ~180MB

# Smallest possible — distroless (Google)
FROM gcr.io/distroless/java17-debian12 # ~220MB, no shell

# OpenJDK (community maintained)
FROM openjdk:17-alpine               # deprecated! avoid
Avoid openjdk: The openjdk images are deprecated. Use eclipse-temurin (Adoptium) or amazoncorretto instead.
Multi-stage — Maven project
Maven multi-stagerecommended
# ── Stage 1: BUILD ──────────────────────────
FROM eclipse-temurin:17-jdk-alpine AS builder
WORKDIR /app

# Copy Maven wrapper + pom.xml FIRST (cache trick)
COPY mvnw .
COPY .mvn .mvn
COPY pom.xml .

# Download dependencies (cached if pom.xml unchanged)
RUN ./mvnw dependency:go-offline -B

# Now copy source code and build
COPY src ./src
RUN ./mvnw package -DskipTests

# ── Stage 2: PRODUCTION ─────────────────────
FROM eclipse-temurin:17-jre-alpine AS production
WORKDIR /app

# Copy ONLY the jar from builder
COPY --from=builder /app/target/*.jar app.jar

EXPOSE 8080
CMD ["java", "-jar", "app.jar"]

WORKDIR /app

COPY --from=builder /app/build/libs/*.jar app.jar

EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
Spring Boot layered JAR (best practice)
What is layered JAR? Spring Boot 2.3+ splits the JAR into layers: dependencies (rarely change), spring-boot-loader, snapshot-dependencies, application (your code — changes most). Each layer = separate Docker layer → faster rebuilds when only your code changes.
Spring Boot layered DockerfileSpring Boot 2.3+
# ── Stage 1: BUILD ──────────────────────────
FROM eclipse-temurin:17-jdk-alpine AS builder
WORKDIR /app
COPY mvnw . 
COPY .mvn .mvn
COPY pom.xml .
RUN ./mvnw dependency:go-offline -B
COPY src ./src
RUN ./mvnw package -DskipTests

# ── Stage 2: EXTRACT layers ─────────────────
FROM eclipse-temurin:17-jdk-alpine AS extractor
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
# Extract layers from the fat JAR
RUN java -Djarmode=layertools -jar app.jar extract

# ── Stage 3: PRODUCTION ─────────────────────
FROM eclipse-temurin:17-jre-alpine AS production
WORKDIR /app

# Copy layers in ORDER: least-changing → most-changing
COPY --from=extractor /app/dependencies/ ./
COPY --from=extractor /app/spring-boot-loader/ ./
COPY --from=extractor /app/snapshot-dependencies/ ./
COPY --from=extractor /app/application/ ./

EXPOSE 8080
# Use JarLauncher instead of java -jar
ENTRYPOINT ["java", "org.springframework.boot.loader.JarLauncher"]
Why this is best: When you change only your code, Docker reuses the cached dependencies layer (the heaviest one). Only the tiny application layer rebuilds → push/pull is 10x faster in CI/CD pipelines like Jenkins and ArgoCD.
JVM memory flags for containers
Container-aware JVM flagsimportant for K8s
CMD ["java", \
  # Respect container memory limits (K8s/Docker)
  "-XX:+UseContainerSupport", \
  # Use max 75% of container memory for heap
  "-XX:MaxRAMPercentage=75.0", \
  # Use G1 GC (best for containers)
  "-XX:+UseG1GC", \
  # Faster startup (skip JIT on rarely used code)
  "-XX:TieredStopAtLevel=1", \
  "-jar", "app.jar"]
Why this matters in Kubernetes: Without UseContainerSupport, JVM reads the HOST machine's RAM (e.g. 32GB) instead of your pod's memory limit (e.g. 512MB) → JVM allocates 8GB heap → pod gets OOMKilled immediately.
Security best practices
Run as non-root usersecurity
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# Create a non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

COPY --from=builder /app/target/*.jar app.jar

# Change ownership
RUN chown appuser:appgroup app.jar

# Switch to non-root user
USER appuser

EXPOSE 8080
CMD ["java", "-jar", "app.jar"]

CMD vs ENTRYPOINT explained
Key difference
# ENTRYPOINT — cannot be overridden easily
ENTRYPOINT ["java", "-jar", "app.jar"]
# docker run myapp              → java -jar app.jar
# docker run myapp --debug      → java -jar app.jar --debug

# CMD — can be overridden by docker run
CMD ["java", "-jar", "app.jar"]
# docker run myapp              → java -jar app.jar
# docker run myapp bash         → bash  (CMD replaced!)

# BEST PATTERN — combine both
ENTRYPOINT ["java", "-jar"]     # fixed command
CMD ["app.jar"]                 # default arg (overridable)
Common Java CMD patterns
java -jar app.jar Basic — run the fat JAR
java -XX:+UseContainerSupport -jar app.jar K8s/Docker memory aware
java -Dspring.profiles.active=prod -jar app.jar Spring Boot prod profile
java -Xms256m -Xmx512m -jar app.jar Fixed heap size (old way)
java org.springframework.boot.loader.JarLauncher Spring layered JAR launcher
ENV variables pattern
Pass config via environment
# Set default ENV in Dockerfile
ENV SPRING_PROFILES_ACTIVE=prod
ENV SERVER_PORT=8080
ENV JAVA_OPTS="-XX:+UseContainerSupport -XX:MaxRAMPercentage=75"

# Use JAVA_OPTS in CMD
CMD ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]

# Override at runtime:
# docker run -e JAVA_OPTS="-Xmx1g" myapp
# kubectl set env deploy/myapp JAVA_OPTS="-Xmx2g"
